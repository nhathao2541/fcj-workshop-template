## Introduction

This chapter summarizes the design decisions of the Smart Notes API from the perspective of two pillars of the AWS Well-Architected Framework: **Cost Optimization** and **Security**. These are not additional steps performed at the end of the project, but design choices that have been considered throughout the workshop, beginning in Chapter 5.2.

---

## Cost Optimization

| Design Decision | Cost Impact |
|---|---|
| AWS Lambda instead of EC2/ECS | No charges while idle—pay only for invocations and execution time (milliseconds) |
| Amazon DynamoDB `PAY_PER_REQUEST` | No cost for unused provisioned capacity |
| DynamoDB TTL for the recycle bin | Expired notes are automatically deleted, preventing unlimited accumulation of obsolete data |
| Amazon S3 instead of storing images in DynamoDB | Amazon S3 is significantly more cost-effective for binary data than storing Base64 data in DynamoDB items |
| Serving the frontend through Lambda (without CloudFront) | Avoids unnecessary CloudFront costs for low traffic, with the option to upgrade later if needed |

**Estimated cost for testing/demo usage:** with a few hundred requests per day, the entire system remains within the **AWS Free Tier** (1 million Lambda invocations per month, 25 GB of DynamoDB storage, and 5 GB of Amazon S3 storage—free for the first 12 months for new AWS accounts).

## Basic Security

### IAM Least Privilege

The Lambda execution role **does not** use `AdministratorAccess`. Instead, AWS SAM generates `DynamoDBCrudPolicy` and `S3CrudPolicy`, allowing access only to the DynamoDB table and Amazon S3 bucket referenced by `!Ref` in the template. As a result, Lambda cannot access any other AWS resources in the account, even if the application is compromised.

### Amazon S3 Block Public Access and Presigned URLs

The image bucket has all four **Block Public Access** settings enabled. Images never have permanent public URLs. Instead, every image is accessed through a **presigned URL**, which expires after 15 minutes and is generated by Lambda using its IAM permissions. Even if an attacker knows the bucket name, they cannot list or directly access objects inside the bucket.

![Amazon S3 Block Public Access](/images/5-Workshop/5.7/s3-block-public-access.png)

### Input Validation in the Service Layer

The `title` field is limited to 100 characters, and the `content` field is limited to 5000 characters. Both fields are required. This validation prevents abnormal payloads from reaching DynamoDB, reducing the risk of injection attacks and oversized requests that increase write costs.

### Securing Amazon API Gateway (Recommended for Production)

The current API is **fully public**, which is suitable for demonstrations and workshops. For production environments, consider adding one of the following:

- **API Key** (`UsagePlan` + `ApiKeyRequired: true`) to reject requests without a valid API key.
- **Amazon Cognito User Pool Authorizer** to authenticate real users, especially when implementing note ownership.

### Avoid Logging Sensitive Data

Current application logs (`console.log`) record only the note `title`, `id`, and file name. The full note `content` is never written to Amazon CloudWatch Logs, reducing the risk of exposing sensitive user data through log access.
